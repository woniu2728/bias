# Bias 项目完成度报告

日期：2026-05-19

## 1. 结论

Bias 当前已经不是“功能原型”或“半成品论坛”，而是一个完成度较高、可运行、具备较强后台与工程化能力的论坛系统。

按不同视角评估，当前完成度可概括为：

- 论坛产品完成度：约 85%
- 平台化/扩展化完成度：约 60%-65%
- 项目综合完成度：约 75%-80%

综合判断：项目已经具备上线级论坛产品的主体能力，但距离“成熟的可扩展论坛平台”仍有一段收口工作要做。

- 阅读仓库文档、路线图、后端核心模块、前端状态层与后台实现
- 检查 registry、Resource API、领域事件、实时广播、安装升级、权限与后台配置代码

## 4. 路线图完成情况判断

结合当前扩展系统路线图与平台现状，当前状态大致如下：

- 阶段 0：大部分已完成
- 阶段 1：主体已完成
- 阶段 2：主体已落地，但还未彻底统一
- 阶段 3：部分完成，仍在演进中
- 阶段 4：局部完成，尚未系统收口
- 阶段 5：明显未完成

具体判断如下。

### 阶段 0：生产稳定性与安全基线

已完成或基本完成：

- 分页参数统一收口，`page/limit` 有规范化逻辑
- 认证已切换为 `HttpOnly Cookie + CSRF` 模式
- Font Awesome 已本地打包，不再依赖外部 CDN
- 回帖楼层并发问题已明显收敛，`PostService` 已通过锁讨论行后再分配楼层号

未完全收口：

- 生产环境对 Redis / 队列 / Channels 仍偏“可降级”，不是“强约束”
- 仍存在真实安全配置风险，测试中出现 JWT key 长度不足的告警

### 阶段 1：扩展核心底座

已明显落地：

- 模块 registry
- 权限注册
- 通知类型注册
- 帖子类型注册
- 搜索过滤器注册
- 讨论排序与列表过滤器注册
- 后台模块中心
- 前端 discussion/header/nav/notification 等注入式渲染

结论：

- 阶段 1 已经不是规划状态，而是已进入可用状态

### 阶段 2：领域事件系统与 Resource API

已落地：

- 事件总线
- 事件监听器
- 时间线事件帖
- Resource 字段和 relationship 注册
- `fields[...]` / `include` 资源查询协议
- 前端 resource store 和 payload merge 机制

仍有不足：

- 部分接口仍是“统一资源能力 + 手工拼 payload”混合模式
- 事件分发仍以内存内同步调用为主，可靠性和隔离性还不够平台级

### 阶段 3 以后

已进入但未完成：

- 前端页面状态模型化程度较高
- 帖流事件语义已开始成立
- 后台模块化管理已较强

未完成部分主要集中在：

- 第三方扩展平台能力
- 更稳定的资源协议统一
- 实时、搜索、长讨论性能的系统性优化
- 主题、国际化、平台生态收口

- 功能覆盖面广，主体论坛能力已齐全
- 后端架构已经出现平台化主干，不是传统的“功能堆接口”
- 前端状态拆分细，说明维护成本已被主动管理
- 后台能力明显强于一般论坛原型项目
- 安装、升级、版本发布链路完整，具备交付意识
- 测试规模可观，前后端都不是低覆盖仓库

## 6. 当前主要问题

### 6.1 平台化尚未完全闭环

虽然 registry、Resource、事件系统已经存在，但当前更接近“具备平台化底座的内置论坛”，还不是成熟的第三方扩展平台。

### 6.2 事件系统可靠性不足

当前领域事件仍是进程内同步分发模型。对论坛产品来说这已经可用，但对平台化来说还不够稳健：

- 监听器失败可能影响主流程
- 缺少更清晰的事务提交后分发边界
- 与队列的关系仍偏松耦合降级，而不是强约束

### 6.3 Resource API 仍未完全统一

当前资源层方向是正确的，但仍存在部分接口手工拼装返回体的问题，这会降低长期一致性。

### 6.4 生产约束还不够硬

项目当前明显兼顾开发体验，因此允许多种降级路径。但真正的生产系统通常需要：

- 更明确的 Redis 必需约束
- 更强的 worker/queue 可用性要求
- 更严格的环境启动检查

### 6.5 安全配置仍有真实风险

测试中出现了 JWT `InsecureKeyLengthWarning`，说明默认或测试配置中存在签名 key 长度不足的问题。这个问题不影响功能可用，但属于必须修复的安全基线项。

### 6.6 少数核心文件已经偏大

以下文件已经显著承担过多职责，后续会成为维护成本来源：

- `apps/core/admin_api.py`
- `apps/core/forum_registry.py`
- `frontend/src/admin/registry.js`
- 若干大型 composable / 视图绑定文件

## 7. 优化优先级建议

### P0：优先立即处理

- 强制生产环境 JWT secret 长度校验
- 对生产环境 Redis / Celery / Channels 增加更强的启动检查和后台告警
- 将关键领域事件调整为事务提交后分发，减少主流程与副作用的直接耦合
- 持续补并发、权限、安全边界测试

### P1：下一阶段主线

- 继续推进 Resource API 统一，减少手工响应拼装
- 把模块启用/禁用、依赖校验、运行时状态做成更完整的生命周期机制
- 拆分超大后台 API 与前端 registry 文件
- 收紧搜索、实时、长讨论列表的性能策略

### P2：平台化收口

- 设计更清晰的第三方扩展接入机制
- 主题、国际化、开发者扩展文档继续完善
- 将“内置模块注册”进一步演进为“可扩展平台协议”

## 8. 建议的下一阶段开发顺序

建议不要再按页面补功能，而应按底座收口顺序推进：

1. 先完成生产安全与运行时约束收紧
2. 再完成事件分发与 Resource API 的一致性改造
3. 然后拆分大文件、压缩维护复杂度
4. 最后推进第三方扩展化、主题化和国际化

## 9. 最终判断

Bias 当前最强的地方，已经不是“做出一个论坛”，而是“已经开始形成自己的论坛平台底座”。

当前短板也很明确：平台化能力已经开头，但还没有完全收口到成熟形态。

因此最准确的项目定位不是：

- “待开发论坛原型”

而是：

- “功能已较完整、架构已进入平台化中段的论坛系统”

如果目标是作为一个可部署论坛产品继续演进，当前代码库已经具备较高价值。

如果目标是对标 Flarum 式可扩展论坛平台，下一阶段重点应放在平台协议、事件可靠性、资源一致性和生产约束，而不是继续堆叠页面功能。
