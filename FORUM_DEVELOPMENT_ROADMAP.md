# Forum Development Roadmap

## 目标

Bias 当前已经具备论坛产品的主体能力。后续开发优先保证正确性、可观测性和性能，再继续扩展新功能。

## 阶段 1：运行链路可观测

目标：让后台能清楚区分“配置存在”和“实际可用”。

- 队列：展示 worker 状态、入队次数、同步执行次数、回退次数、最近任务和最近错误。
- 队列：提供指标重置入口，方便部署后手动验证。
- 邮件：测试邮件、验证邮件、密码重置邮件都通过统一队列入口。
- 缓存：后台清理缓存后同步清理运行时设置缓存。

验收标准：

- 队列关闭时任务同步执行，指标记录同步次数。
- 队列开启且 Redis worker 可用时任务入队，指标记录入队次数。
- worker 不可用或入队失败时同步回退，后台显示失败信息。
- 管理员可以在仪表盘重置队列运行指标。

## 阶段 2：规则一致性

目标：列表、详情、搜索、审核队列对内容可见性的判断一致。

- 统一讨论和帖子的 `approved / pending / rejected / hidden` 可见性。
- 删除讨论或帖子后重算讨论统计、作者统计、标签统计。
- 隐藏、恢复、审核、拒绝等操作全部走 service 层。
- 补充搜索、列表、详情、审核流的回归测试。

验收标准：

- 普通用户不会通过搜索看到无权访问内容。
- 作者能看到自己的待审核和被拒内容。
- 管理员能看到需要处理的审核内容。
- 删除最后回复后，讨论列表的最后回复信息正确。

## 阶段 3：性能治理

目标：把高频读请求从全表扫描和重复统计中解放出来。

- 搜索：减少重复 `count()`，消除讨论摘要 N+1。
- 搜索：生产环境基于 PostgreSQL 全文索引升级。
- 标签：标签列表不在递归序列化中重复计算权限。
- 标签：统计重算从读接口移到写后任务或后台命令。
- 通知：未读数从每次 count 优化为缓存或计数字段。
- 浏览量：从每次访问写数据库优化为缓存累积后异步落库。

验收标准：

- 标签列表和搜索请求 SQL 数量明显下降。
- 大量通知用户打开页面时不反复全量统计未读数。
- 热门讨论访问不会因为浏览量写入造成数据库热点。

## 阶段 4：UI 与交互收口

目标：把已有页面沉淀成稳定设计系统。

- 统一按钮、表单、徽章、状态块、确认弹窗、统计块。
- 清理未使用页面和重复样式。
- 后台高风险操作统一确认和结果反馈。
- Composer 错误状态分层展示：上传、预览、草稿、提交。

验收标准：

- 前后台基础控件视觉和状态一致。
- 删除、封禁、审核、清理等操作都有明确确认和反馈。
- 移动端列表、详情、后台表单没有文字溢出或操作遮挡。

## 阶段 5：产品化增强

目标：补齐长期运营能力。

- 管理员操作审计日志。
- 站点公告和维护提示增强。
- 通知偏好设置。
- SEO meta 和站点分享信息。
- 更完整的对象存储适配和上传策略。
- Refresh token 迁移到 HttpOnly Cookie，降低令牌暴露风险。

## 当前开发顺序

1. 完成队列指标重置入口。
2. 优化标签列表权限计算。
3. 优化搜索 count 和摘要查询。
4. 优化通知未读数。
5. 建立后台确认弹窗和审计日志基础。
