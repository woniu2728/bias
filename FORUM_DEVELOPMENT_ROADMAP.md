# Bias 开发路线图

更新时间：2026-05-19

## 1. 当前项目定位

Bias 现阶段不再是“论坛原型”，而是一个已经具备主体论坛能力、并且开始形成平台化底座的论坛系统。

从当前代码实际状态判断：

- 论坛产品能力已较完整
- 后台与工程化能力明显强于一般中小型论坛项目
- 扩展化、资源协议、事件驱动已经开始落地
- 但测试稳定性、生产约束、平台一致性和可扩展边界仍未完全收口

因此，后续开发不应再以“继续补页面功能”为主，而应切换为：

- 先收口质量与运行时边界
- 再统一平台底座
- 最后再扩展生态能力

这份路线图将作为后续开发的唯一主线。

## 2. 总目标

把 Bias 从“完成度较高的论坛应用”推进为“可稳定部署、可持续迭代、具备清晰扩展边界的论坛平台”。

后续开发遵循四个原则：

1. 先保证测试与运行时稳定，再继续叠加能力
2. 优先统一后端事件、资源协议和前端状态边界
3. 优先降低维护复杂度，而不是继续放大大文件和手工接口
4. 所有平台化抽象必须服务论坛主场景，不做脱离业务的空泛设计

## 3. 当前问题总结

当前仓库最明显的问题不是“功能缺失”，而是“收口不完整”。

### 3.1 测试体系尚未完全稳定

- 前端测试已经具备较好稳定性
- 后端测试规模不小，但仍存在失败用例
- 部分测试运行过程中会触发 Redis / Celery / 搜索等环境相关告警或失败

这意味着项目已经有测试文化，但还没有完全达到“回归结果可靠”的状态。

### 3.2 生产运行时约束不够硬

- Redis / Celery / Channels 仍以可降级为主
- 默认配置仍保留开发型 secret / JWT key 兜底
- 启动阶段对生产风险的阻断能力不够强

这类问题不影响本地开发，但会影响生产质量下限。

### 3.3 事件系统与资源协议未完全统一

- 领域事件总线已经存在
- `transaction.on_commit` 分发能力已经存在
- 但部分流程仍有直接同步分发
- Resource API 已有基础设施，但接口层仍存在手工拼装 payload

也就是说方向是对的，但平台协议还没有收口成统一约束。

### 3.4 少数核心文件已经过大

当前以下文件已经成为维护风险点：

- `apps/core/admin_api.py`
- `apps/core/forum_registry.py`
- `apps/discussions/services.py`
- `apps/posts/services.py`
- `frontend/src/admin/registry.js`

继续在这些文件上堆功能，会明显拉高后续修改成本和回归风险。

### 3.5 平台化边界仍偏“内置模块注册”

虽然已经有 registry、resource、event listener、前端注入点，但当前更接近：

- “内置模块平台化”

还不是：

- “成熟的第三方扩展平台”

所以接下来的重点不是新增更多注册点，而是把已有机制做实、做稳、做一致。

## 4. 新路线图总顺序

后续开发严格按以下顺序推进：

1. 阶段 A：测试与质量收口
2. 阶段 B：生产安全与运行时约束收口
3. 阶段 C：事件系统与资源协议统一
4. 阶段 D：核心模块拆分与维护性治理
5. 阶段 E：扩展平台边界完善
6. 阶段 F：性能、生态与平台收尾

原则：

- 未完成前一阶段的最低验收标准，不进入后一阶段主线
- 如出现严重回归，优先回到阶段 A/B 处理

## 5. 阶段 A：测试与质量收口

### 目标

先把项目从“测试很多”推进到“测试结果可靠、可作为迭代护栏”。

### 优先级

P0

### 任务

- 修复当前 `pytest` 失败用例
- 修复当前 `python manage.py test` 中的失败用例
- 梳理测试中对 Redis、Celery、搜索、上传等外部依赖的隐式耦合
- 将环境相关能力在测试中统一切换为明确的 mock、fake 或测试配置
- 为高风险路径补测试：
  - 权限边界
  - 审核流
  - 举报处理
  - 回帖并发编号
  - 搜索可用性降级
  - 事件触发后的副作用链路
- 统一后端测试分层：
  - 纯 service 测试
  - API 集成测试
  - 运行时/命令测试

### 验收标准

- `pytest` 全量通过
- `python manage.py test` 全量通过
- 测试运行不再依赖真实 Redis / worker 可用性
- 高风险主流程具备明确回归保护

## 6. 阶段 B：生产安全与运行时约束收口

### 目标

把当前“可开发、可降级”的运行方式，收口到“生产有边界、有阻断、有告警”。

### 优先级

P0

### 任务

- 强制校验生产环境 `SECRET_KEY` / `JWT_SECRET_KEY` 长度与安全性
- 为生产模式增加启动自检：
  - Redis 可用性
  - Celery broker 可用性
  - Channels 配置状态
  - 关键站点配置完整性
- 对后台运行状态页做收口，明确展示：
  - queue 状态
  - worker 状态
  - cache 状态
  - websocket 状态
  - 搜索状态
- 明确 dev / test / production 三套运行时约束，不再使用模糊兜底
- 收紧默认配置中不适合生产的行为：
  - 默认密钥
  - 本地邮件后端
  - memory queue / in-memory channel layer

### 验收标准

- 生产环境缺少关键依赖时，系统可明确告警或拒绝启动
- 后台能展示关键基础设施状态
- 默认配置不再给生产误用留下明显漏洞

## 7. 阶段 C：事件系统与资源协议统一

### 目标

把当前已经存在的“事件驱动 + Resource API”从可用状态推进到统一状态。

### 优先级

P0-P1

### 任务

- 统一领域事件分发规则：
  - 所有会影响副作用的核心事件，统一在事务提交后分发
  - 禁止继续新增流程内直接同步 `dispatch`
- 梳理并统一事件来源：
  - discussion 审核
  - post 审核
  - 用户封禁/解封
  - 标签统计刷新
  - 通知触发
  - 实时广播
- 建立“核心事件清单”和“监听器责任边界”
- 继续推进 Resource API 统一：
  - 优先收口 discussion / post / user / tag / notification 主资源
  - 减少 endpoint 手工拼装字段
  - 统一 `fields[...]` 和 `include` 的使用方式
- 前端 resource store 与页面数据绑定继续统一，减少页面级重复转换

### 验收标准

- 核心主流程的副作用不再依赖同步散落调用
- 主资源接口的结构明显统一
- 新增接口默认优先走 resource registry，而不是手工序列化

## 8. 阶段 D：核心模块拆分与维护性治理

### 目标

降低大文件和复合职责带来的维护风险，为后续持续开发释放空间。

### 优先级

P1

### 任务

- 拆分 `apps/core/admin_api.py`
  - 按 dashboard、users、permissions、flags、approval、settings、operations 分域
- 拆分 `frontend/src/admin/registry.js`
  - 按 dashboard、page config、copies、actions、page meta 分模块
- 梳理并拆分超大 service 文件：
  - `apps/discussions/services.py`
  - `apps/posts/services.py`
- 收口模型层遗留逻辑
  - 避免模型 `save()` 中保留会绕过 service 约束的关键业务逻辑
- 减少跨模块隐式耦合，明确：
  - service 层职责
  - 事件监听职责
  - registry 注册职责

### 验收标准

- 超大文件被拆到可维护规模
- 核心业务约束集中在 service 层
- 后续新增功能不再默认堆进历史大文件

## 9. 阶段 E：扩展平台边界完善

### 目标

把当前“内置模块注册化”进一步推进为“平台边界清晰的扩展体系”。

### 优先级

P1-P2

### 任务

- 完善模块生命周期定义：
  - register
  - bootstrap
  - ready
  - optional disable / teardown
- 明确模块依赖校验、状态展示、配置入口、健康状态
- 继续完善前端注入点：
  - header
  - discussion actions
  - post actions
  - composer extension
  - admin navigation
  - notification renderer
- 为开发者文档补充“如何新增一个模块”的标准流程
- 评估并设计第三方扩展接入的最小协议

### 验收标准

- 内置模块已经按照统一生命周期管理
- 新模块接入流程清晰且文档化
- 前后端扩展接口具备一致的接入方式

## 10. 阶段 F：性能、生态与平台收尾

### 目标

在底座稳定后，再系统处理性能和生态层能力，而不是提前做局部优化。

### 优先级

P2

### 任务

- 长讨论页性能治理
- 列表与搜索深分页策略优化
- 实时广播负载控制与降噪
- 搜索索引刷新策略优化
- 主题能力继续推进
- 国际化能力继续推进
- 开发者文档、部署文档、平台文档继续完善

### 验收标准

- 长讨论、搜索、实时场景有明确性能策略
- 平台使用文档达到可交付水平
- 论坛产品与平台底座都具备持续演进基础

## 11. 明确不作为当前主线的事项

以下事项暂不作为当前阶段主线，除非与 A/B/C/D 阶段直接相关：

- 继续大规模新增前台页面功能
- 继续扩张后台页面数量
- 提前做主题市场或在线插件市场
- 提前做复杂微服务化拆分
- 为抽象而抽象的通用平台设计

原因很明确：

- 当前项目的主要问题不是功能不够，而是底座和质量收口不够

## 12. 后续执行规则

从本路线图起，后续开发按以下规则执行：

1. 新任务默认先判断属于哪个阶段
2. 如任务不属于当前阶段主线，原则上降级优先级
3. 涉及底座改造时，必须同步补测试
4. 涉及事件、资源协议、运行时约束的修改，优先保证一致性，不接受局部特殊写法继续扩散
5. 新增功能前，先判断是否会继续放大已有大文件；如果会，优先拆分后再开发

## 13. 当前建议的直接下一步

下一步开发建议从以下顺序开始：

1. 先定位并修复当前 `pytest` 与 `manage.py test` 失败用例
2. 然后收紧生产安全配置和运行时健康检查
3. 再统一事件分发边界和 Resource API 主资源
4. 然后拆分后台 API 与前端 admin registry

这是当前最稳妥、收益最高的推进路径。
